## Purpose

Wire the prompt registry into the application runtime container and expose a narrow API-edge dependency for retrieving the already-wired registry.

This task makes the prompt registry available to later modules such as RAG and to the prompt inspection API introduced in T-1004. It does **not** add prompt inspection endpoints.

## Implementation requirements

### Wiring module

Create:

```text
app/core/wiring/prompts.py
```

The wiring module must:

- compute the package-local prompt library path for `app/prompts/library/`
- instantiate the T-1001 filesystem registry adapter
- return it as the `PromptRegistry` port type
- load/validate prompt metadata during startup so malformed shipped prompts fail fast
- contain construction/configuration only
- contain no business logic

The wiring module may import the concrete `app.prompts.adapters.filesystem_registry.FileSystemPromptRegistry` because `app.core.wiring.*` is the approved composition surface.

### Container

Update the application container to expose a port-typed prompt registry field, for example:

```python
prompt_registry: PromptRegistry
```

Preserve the existing container style.

The container must remain a small typed record. Do not convert it into a registry, service locator, or factory-of-factories.

### Lifespan integration

Update application startup wiring so the prompt registry is constructed and stored in the container alongside other already-wired application dependencies.

Respect the existing lifespan ordering.

If the project sets `app.state.ready = True`, ensure readiness is only marked true after prompt registry wiring succeeds.

If the prompt registry does not hold async resources, do not add unnecessary shutdown hooks.

### API dependency

Create:

```text
app/prompts/deps.py
```

Provide a narrow dependency that retrieves the already-wired prompt registry from the application container, for example:

```python
def get_prompt_registry(request: Request) -> PromptRegistry: ...
```

This dependency may depend on FastAPI/Starlette request types because it is an API-edge dependency module.

It must not construct the registry, load YAML, create Jinja environments, or import the concrete filesystem adapter.

## Tests required

Add or update tests covering:

- prompt wiring builds a `PromptRegistry`
- the wired registry can list the shipped prompt from T-1002
- the application container exposes the registry as a port-typed dependency
- application lifespan succeeds with prompt wiring enabled
- invalid prompt YAML in the package-local library causes startup/wiring failure before the application is marked ready
- generic lifespan tests remain isolated and do not require live Redis/Postgres/external HTTP beyond what those tests already intentionally cover

Do not add API route tests in this task. API route tests belong to T-1004.

## Acceptance criteria

- The application container exposes the prompt registry as a `PromptRegistry` port, not a concrete filesystem registry.
- `app.core.wiring.prompts` constructs the filesystem registry.
- `app.prompts.deps` retrieves the already-wired registry from `request.app.state.container`.
- API-edge dependency code does not construct registries, load YAML files, or create Jinja environments.
- Invalid prompt YAML in the package-local library causes startup/wiring failure before the application is marked ready.
- `app/core/wiring/prompts.py` owns prompt registry construction.
- The prompt registry is added to the application container as a port-level dependency.
- `app/prompts/deps.py` retrieves the already-wired registry from the container.
- `app.main` does not import `app.infrastructure` or prompt concrete adapters.
- `app.api` does not construct prompt infrastructure or prompt registries.
- No prompt inspection routes are added yet.
- `make lint`, `make typecheck`, `make test`, and `uv run lint-imports` pass.
