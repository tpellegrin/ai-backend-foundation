## Implementation requirements

### API module

Create:

```text
app/prompts/api.py
```

Declare a module router with no prefix. The `/prompts` prefix must be applied at the API v1 mount site.

Endpoints:

```text
GET /prompts
GET /prompts/{prompt_id}/{version}
```

Effective routes after API v1 mounting:

```text
GET /api/v1/prompts
GET /api/v1/prompts/{prompt_id}/{version}
```

The API must retrieve the registry through `app.prompts.deps.get_prompt_registry` from T-1003.

The API must use explicit response models so FastAPI validates, documents, and filters response data.

Recommended response shape:

```python
class PromptSummaryResponse(BaseModel):
    id: str
    versions: list[str]

class PromptDetailResponse(BaseModel):
    id: str
    version: str
    description: str
    owner: str
    input_schema: str
    output_schema: str | None
    confidential: bool
```

`GET /prompts` should return grouped prompt summaries:

```json
[
  {"id": "rag_answer", "versions": ["1.0.0"]}
]
```

`GET /prompts/{prompt_id}/{version}` should return prompt metadata. The S10 inspection API does not return raw template bodies.

### Authorization

These endpoints are restricted to administrative users. Use `Annotated[AuthenticatedUser, Depends(require_scope("prompts:read"))]` (the dependency established in `T-911`) to protect these routes.

### Router registration

Update:

```text
app/api/v1.py
```

Append the prompt router registration to the v1 router builder using the existing project convention.

The module router in `app/prompts/api.py` must have no prefix; the prefix is applied at mount site:

```python
router.include_router(prompts.api.router, prefix="/prompts", tags=["prompts"])
```

Do not reorder existing routers.

### Consolidated shipped-prompt verification

Add:

```text
app/prompts/tests/test_registry.py
```

This test file should parameterize over all prompts present in `app/prompts/library/` and verify that each shipped prompt:

- loads through the registry
- has a resolvable input schema
- has a resolvable output schema when declared
- renders successfully with representative valid sample input using the representative samples provided in the test code
- can validate representative output when an output schema is declared

For T-1004, it is acceptable to include explicit representative samples for the shipped `rag_answer` prompt. Do not invent a large prompt sample framework.

## Tests required

API tests must cover:

- unauthenticated request is rejected according to the existing project convention
- authenticated non-admin/non-scope request is rejected if scope authorization exists
- authorized/admin request to `GET /api/v1/prompts` returns the shipped prompt summary
- authorized/admin request to `GET /api/v1/prompts/rag_answer/1.0.0` returns metadata
- unknown prompt ID/version returns the project's standard not-found error
- both routes appear in OpenAPI

Registry verification tests must cover:

- all shipped prompts load through the wired registry
- `rag_answer` renders with valid representative input
- `RagAnswerOutput` validates a representative answer

## Acceptance criteria

- `app/prompts/api.py` defines read-only inspection routes only.
- API handlers retrieve the already-wired registry through `app.prompts.deps`.
- API handlers do not construct registries, load YAML, or create Jinja environments.
- Response models are explicit and do not leak internal implementation objects.
- Prompt router registration is append-only in `app/api/v1.py`.
- Endpoints are protected by `require_scope("prompts:read")` from T-911.
- Consolidated registry tests cover every shipped prompt in the prompt library.
- No prompt mutation endpoint is added.
- The S10 inspection API does not return raw template bodies.
- No RAG orchestration, LLM call, provider integration, or vector-store logic is added.
- `make lint`, `make typecheck`, `make test`, and `uv run lint-imports` pass.
