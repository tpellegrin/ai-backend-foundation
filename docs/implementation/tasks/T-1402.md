Purpose: the worker boots through the **same** composition root as the API.

Depends on: T-1401, T-504.

Allowed files: `app/core/wiring/queue.py`, `app/worker.py`, `app/core/container.py` (append-only), `app/core/lifespan.py` (append-only), `tests/integration/test_worker_wiring.py`.

Forbidden: defining a second container or settings reader for the worker; opening a second Redis client or DB engine outside the shared lifespan hooks. Edits to `container.py` and `lifespan.py` are strictly append-only per rules.md Â§1 "Append-only edits to shared assembly files".

Implementation requirements

- `app/worker.py` is the sole worker entrypoint. It re-exports `WorkerSettings` from `app.core.wiring.queue` (`from app.core.wiring.queue import WorkerSettings as WorkerSettings`) so that `arq app.worker.WorkerSettings` is the stable public command line. `app/worker.py` contains no business logic â€” it is a thin ASGI-independent entrypoint that only exposes `WorkerSettings`. It must not import provider SDKs, not construct a Container directly, and not read env vars other than through the shared settings module.
- `app/core/wiring/queue.py` defines `WorkerSettings`: Arq config with `functions = [documents.ingestion.ingest_document]` (function objects registered here), `on_startup`/`on_shutdown` hooks that initialize the **same** `Container` used by the API via the shared `app.core.lifespan` startup/shutdown routines and tear it down symmetrically. It also builds the `TaskQueue` port adapter (Arq submitter) for the API side.
- `make worker` runs `arq app.worker.WorkerSettings`.
- Job context exposes `container` so handlers can resolve services (`ctx["container"]`).
- Append to `app/core/container.py`: a `task_queue: TaskQueue` field (Protocol-typed).
- Append to `app/core/lifespan.py`: startup/shutdown hooks that build and tear down the Arq submitter (`TaskQueue`) shared by the API. If the worker requires its own additional startup steps (e.g. warming the container), they live inside `WorkerSettings.on_startup` in `queue.py`, not by rewriting the shared lifespan. Register a queue readiness probe on the `ProbeRegistry` (per rules.md Â§1 incremental-probe rule).

Tests required (integration): worker startup populates the container with the same field types as the API (assert types match Protocols); `app.worker.WorkerSettings` is importable and is identical to `app.core.wiring.queue.WorkerSettings`; queue readiness probe is registered.

Acceptance: API and worker cannot diverge in wiring; a single import path (`app.worker.WorkerSettings`) drives `arq`.

Otherwise standard.
