# Task T-709 — Lifespan readiness ordering fix

> Section: **S07 — Infrastructure base**
>
> Before you start, read [`../rules.md`](../rules.md).
> After you finish, self-check against [`../review.md`](../review.md).

---

Purpose: fix a readiness-flag race in the application lifespan. The current
`app/core/lifespan.py` sets `app.state.ready = True` **before** the Redis
client is constructed, the `Cache` port is bound into the `Container`, and
the `RedisProbe` is registered on the `ProbeRegistry`. As a result,
`/readyz` can report `200 OK` while the Redis-backed subsystems are still
uninitialized. This task moves the readiness flip to the last step of
successful startup, so `/readyz` only returns success once all wired
resources have been constructed and all readiness probes are installed.

Depends on: T-708.

Allowed files:
- `app/core/lifespan.py` (narrow edit for ordering only)
- `tests/integration/test_lifespan.py`
- `docs/implementation/roadmap.md`

Forbidden changes:
- Do not add new fields to `app/core/container.py`.
- Do not wire `RateLimiter`, `IdempotencyStore`, or the shared HTTP client
  into the `Container` or `lifespan`. Those belong to their first
  consumer's task.
- Do not create or modify any ADR.
- Do not modify `.importlinter`.
- Do not add re-exports to `app/core/wiring/__init__.py`.
- Do not modify any infrastructure adapter, port, or provider code.
- Do not introduce `use_cases/`, a global `app.ports`, or any strict Clean
  Architecture folders.
- Do not import `app.infrastructure.*` outside `app.core.wiring.*`.
- Do not introduce business logic in `lifespan.py`.

Implementation requirements:
- In `app/core/lifespan.py`, keep `app.state.ready = False` at the start
  of the lifespan.
- Delete the premature `app.state.ready = True` currently set after
  `_on_startup`. Move it so it is the **last statement before `yield`**,
  after Redis client construction, cache binding, and `RedisProbe`
  registration.
- The `try/finally` block that owns Redis cleanup must remain; only the
  ordering of the readiness flag changes inside that block.
- Do not alter shutdown ordering: `app.state.ready = False` on the way
  out must still precede `_on_shutdown` and Redis client `aclose()`.
- No new imports. No new hooks. No new probes.

Tests required:
- `tests/integration/test_lifespan.py`:
    - Add an assertion that during the lifespan window `app.state.ready`
      is `True` **and** the `ProbeRegistry` contains a probe named
      `"redis"`.
    - Add a test that stubs `setup_redis_client` to raise; assert that
      `app.state.ready` is never observed as `True` and that `/readyz`
      does not report ready. The lifespan may propagate the error; the
      invariant is that readiness is not flipped when startup fails.
    - Preserve the existing pattern of mocking Redis/DB wiring via
      `monkeypatch`; do not require a live Redis or Postgres instance.

Acceptance:
- `app.state.ready` is set to `True` only after all currently-wired
  resources (DB, storage, cache) are initialized and their probes
  registered.
- Existing integration tests continue to pass without a live Redis.
- `lint-imports`, `ruff`, `mypy --strict`, unit and integration tests
  all pass.

Commands: standard four.

Common failure modes:
- Leaving both the old and the new `ready = True` line (double flip).
- Moving the readiness flip *after* `yield` (would make the app never
  report ready).
- Introducing a live-Redis requirement into generic lifespan tests.

Review checklist:
- Single `app.state.ready = True` assignment, positioned immediately
  before `yield`, after Redis wiring and `RedisProbe` registration.
- `Container` shape unchanged.
- No new imports or fields.
- No behavior change on shutdown.
- Tests assert the ordering invariant, not just the endpoint response.
