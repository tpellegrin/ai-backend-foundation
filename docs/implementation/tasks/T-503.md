# Task T-503 — `app/api/v1.py` router mount point

> Section: **S05 — App factory + API edge**
>
> Before you start, read [`../rules.md`](../rules.md) (global rules, stop signals, allowed-files discipline, command discipline, lower-complexity model instructions).
> After you finish, self-check against [`../review.md`](../review.md) (final review checklist, architecture compliance).

---


Purpose: a single place that mounts each module's router under `/api/v1`.

Depends on: T-501.

Allowed files: `app/api/v1.py`.

Forbidden: do not put business logic here; do not import `app.<module>.service`.

Implementation requirements

- `def build_v1_router() -> APIRouter:` returns an **empty** `APIRouter` mounted at prefix `/api/v1` — this task ships the mount point only.
- **The `/api/v1` router is incremental.** Each later module task (`auth`, `users`, `prompts`, `governance`, `documents`, `rag`) appends its own router — via `router.include_router(<module>.api.router, prefix=..., tags=...)` — in the same PR that introduces that module's `api.py`. T-503 must not import any module's `api.py` (none exist yet); it must not enumerate module routers as a hard requirement.
- A task may not reference a module router that has not yet been added by its own task. This mirrors the incremental-`Container` rule (`rules.md` §1).

Tests required: `build_v1_router()` returns an `APIRouter` mounted at prefix `/api/v1` with zero routes at this phase. Regression test for the fully-mounted router (once all modules are added) lives in T-507.

Acceptance criteria: `/api/v1` mount point exists; adding a module router later requires only an `include_router` edit here plus that module's `api.py` — added in the module's own task.

Commands: standard four.

Common failure modes: importing services or persistence here.

Review checklist: imports limited to per-module `api`, single mount point.
