## Purpose

Introduce the minimal prompt foundation for Phase 2:

- a module-owned `Prompt` domain object
- a module-owned `PromptRegistry` port
- a filesystem/package-backed registry implementation for trusted project-owned YAML prompt definitions
- deterministic Jinja2 rendering with input schema validation

This task establishes `app.prompts` as the owner of prompt definitions and rendering behavior. It does **not** add shipped production prompts, application container wiring, API routes, RAG logic, LLM calls, or provider integrations.

## Implementation requirements

### Domain model

Create a `Prompt` domain object in `app/prompts/domain.py`.

Use a small immutable typed shape (a frozen dataclass).

Required fields:

- `id: str`
- `version: str`
- `description: str`
- `owner: str`
- `template: str`
- `input_schema: type[app.shared.pydantic.BaseSchema]`
- `output_schema: type[app.shared.pydantic.BaseSchema] | None`
- `confidential: bool = False`

Version should be treated as an opaque string at this stage. Do not introduce a semver dependency or version parser unless an existing project convention already exists.

The domain object must not import Jinja2 or YAML.

### Port

Create `PromptRegistry` in `app/prompts/ports.py` as a `Protocol`.

Required methods:

```python
from app.prompts.domain import Prompt
from app.shared.pydantic import BaseSchema

class PromptRegistry(Protocol):
    def list(self) -> Sequence[Prompt]: ...
    def get(self, prompt_id: str, version: str) -> Prompt: ...
    def render(self, prompt_id: str, version: str, inputs: BaseSchema) -> str: ...
```

The port should expose prompt-domain concepts only. It must not expose Jinja2, YAML, filesystem paths, importlib details, or provider-specific objects.

### Filesystem/package registry adapter

Create the concrete implementation in:

```text
app/prompts/adapters/filesystem_registry.py
```

The adapter may import Jinja2, YAML parsing utilities already used by the project, `pydantic`, `importlib`, and standard-library filesystem/path helpers.

The adapter should be named clearly, for example:

```python
FileSystemPromptRegistry
```

It must:

- load trusted project-owned YAML prompt definitions from a configured prompt library path
- scan only `*.yaml` files from that path
- parse required metadata
- resolve `input_schema` and `output_schema` references from explicit schema path strings
- validate that resolved schemas are subclasses of `app.shared.pydantic.BaseSchema`
- fail fast when a YAML file is malformed or references an invalid schema
- render templates with a dedicated Jinja2 `Environment`
- use `StrictUndefined`
- explicitly set `autoescape=False` because these are LLM prompt text templates, not HTML templates
- validate render inputs against the prompt's declared `input_schema` before rendering
- pass `model_dump()` output into Jinja rather than raw model objects
- raise project-controlled errors for unknown prompt IDs/versions, invalid YAML, invalid schema references, and render failures

Schema resolution may use `importlib` for declared prompt schema paths only. This is allowed here as data-driven schema resolution for trusted project-owned prompt metadata. It must not be used to hide forbidden architectural imports or bypass import-linter boundaries.

Accepted schema reference format:

```text
module.path:ClassName
```

Example:

```text
app.prompts.library.rag_answer_v1:RagAnswerInput
```

### Jinja2 behavior

Use a central `Environment` instance owned by the registry adapter. Do not create ad hoc template environments per render call unless tests show there is a correctness reason.

Rendering must fail if the template references an input field that is not present after schema validation.

The registry handles trusted repository templates only. User-authored or database-authored templates are out of scope.

### Error handling

Use existing project error conventions if available.

If no prompt-specific errors exist yet, add minimal prompt errors inside the allowed prompt files only. Do not modify shared error modules unless this task is explicitly patched to allow it.

Errors should be safe and should not expose raw filesystem internals unnecessarily.

## Tests required

Add tests in:

```text
app/prompts/tests/test_filesystem_registry.py
```

Tests must use temporary test prompt files and local test schema classes/modules as needed. They must not depend on the production prompt library introduced by T-1002.

Cover at least:

- registry loads a valid YAML prompt
- `list()` returns available prompts
- `get()` returns the requested prompt by ID/version
- unknown prompt lookup raises a controlled project error
- valid Pydantic input renders deterministically
- missing template variable fails because of `StrictUndefined`
- invalid input fails with `pydantic.ValidationError`
- invalid schema path fails at registry load time
- missing required YAML field fails at registry load time
- `output_schema` may be omitted or null

## Acceptance criteria

- `Prompt` is defined in `app/prompts/domain.py` and contains no Jinja2/YAML/provider imports.
- `PromptRegistry` is defined in `app/prompts/ports.py` as a protocol.
- Input and output schemas resolved by the registry must be subclasses of `app.shared.pydantic.BaseSchema`.
- The concrete filesystem/package registry implementation lives under `app/prompts/adapters/`.
- `FileSystemPromptRegistry` loads trusted package-local YAML prompt files from `app/prompts/library/`.
- Jinja2 rendering uses a central `Environment` configured with `StrictUndefined`.
- Autoescape is explicitly disabled for trusted LLM prompt text.
- Trusted schema references use `module.path:ClassName`.
- Invalid or missing YAML fields fail with a project-controlled error.
- `.importlinter` is updated to remove parentheses from `app.prompts` in the relevant layer/independence contracts.
- Jinja2 is isolated to the adapter implementation and direct adapter tests.
- YAML parsing is isolated to the adapter implementation and direct adapter tests.
- Rendering validates against the declared input schema before template rendering.
- Rendering uses `StrictUndefined`.
- The registry can resolve declared Pydantic schema references from trusted prompt YAML metadata.
- The registry does not require FastAPI, application container wiring, or infrastructure adapters.
- Tests pass with temporary prompt fixtures.
- `make lint`, `make typecheck`, `make test`, and `uv run lint-imports` pass.

---

## Notes

This task intentionally creates a module-local adapter under `app.prompts.adapters` instead of `app.infrastructure` because the registry reads trusted package-local prompt assets and does not integrate with an external driven resource such as a database, Redis, object storage, or provider SDK.

If prompt definitions later move to a database, object storage, remote registry, or admin-authored prompt-management service, that future adapter should be evaluated as infrastructure and wired through `app.core.wiring.*`.
