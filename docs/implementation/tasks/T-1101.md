Purpose: domain types, ports, observation record, and provider-selection router.

Naming note (per forward-audit patch P-9): the file `app/llm/router.py` implements a **provider/model-selection** router — it maps an `intent` to a concrete model id / adapter binding. It is **not** a FastAPI `APIRouter`. To avoid confusion with FastAPI vocabulary, the exported Protocol is named `ModelRouter` and the default implementation is `DefaultModelRouter`; no symbol named `router` is exported from this module. If a future task adds a FastAPI router at `app/llm/api.py`, that file's `router` symbol is unrelated to this one.

Depends on: T-203, T-401.

Allowed files: `app/llm/__init__.py`, `app/llm/{domain,ports,observability,router}.py`, tests.

Forbidden: any provider SDK import; any HTTP call. No knowledge of OpenAI.

Implementation requirements

- `domain.py`: `ChatMessage(role: Literal["system","user","assistant","tool"], content: str)`; `ChatRequest(model: str, messages: tuple[ChatMessage, ...], temperature: float = 0.0, max_tokens: int | None, response_format: Literal["text","json"] = "text")`; `ChatResponse(content: str, tokens_in: int, tokens_out: int, finish_reason: str, raw_id: str)`; `UpstreamError(AppError)`.
- `ports.py`: `ChatModel(Protocol)` async `complete(req) -> ChatResponse`, `stream(req) -> AsyncIterator[str]`; `ModelRouter(Protocol)` `pick(intent: str) -> str` (returns model id). The Protocol admits multiple models in principle, but see the Phase 2 scope note below.
- `observability.py`: `LLMCallObservation` frozen dataclass with the **11 mandatory fields** (provider, model, prompt_id, prompt_version, tokens_in, tokens_out, latency_ms, cost_usd, status, request_id, tenant_id). `record(observation)` writes structlog event `llm.call` and attaches the same attributes to the active OTel span (span name `llm.chat`).
- `router.py`: `DefaultModelRouter(intent_map: Mapping[str, str])` returning the configured model.
- **Phase 2 scope (per forward-audit patch P-10):** in Phase 2 there is exactly one chat model, sourced from `settings.openai.chat_model`. The wiring layer (T-1212) constructs `DefaultModelRouter({intent: settings.openai.chat_model for intent in KNOWN_INTENTS})` — every intent resolves to the same model id. `intent_map` remains a `Mapping[str, str]` to preserve the shape, but multi-model routing (per-intent model selection, per-tenant overrides, provider failover) is **deferred to Phase 3** and must not be introduced here. `settings.openai.chat_model` stays a single `str` in T-301; do not widen it.

Tests required: ChatRequest immutability; observation serializes with all 11 fields; router picks expected model.

Acceptance: no SDK imports anywhere in `app/llm/`.

Otherwise standard.
