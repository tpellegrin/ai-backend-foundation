# Task T-405 ‚Äî Health endpoints `/healthz` `/readyz` `/livez`

> Section: **S04 ‚Äî Observability**
>
> Before you start, read [`../rules.md`](../rules.md) (global rules, stop signals, allowed-files discipline, command discipline, lower-complexity model instructions).
> After you finish, self-check against [`../review.md`](../review.md) (final review checklist, architecture compliance).

---


Purpose: define liveness/readiness endpoint shapes and a pure probe registry. **Runtime wiring is out of scope**: T-504 bootstraps an **empty** `ProbeRegistry` on the `Container` and owns the `app.state.ready` readiness flag via the lifespan; real probes are appended by their own wiring tasks alongside their `Container` field (T-701 DB, T-702/T-708 Redis/cache, T-1402 queue, T-1503 vector store). This task ships pure, framework-thin building blocks ‚Äî no lifespan hooks, no `core.wiring` imports, no `app.state` reads.

Depends on: T-404.

Allowed files: `app/observability/__init__.py`, `app/observability/health.py`, `app/observability/tests/test_health.py`.

Implementation requirements

- A `Probe` type (Protocol or callable alias) representing an async health check returning a small status object (name + ok/degraded/error + optional latency_ms).
- A pure `ProbeRegistry` container that composes zero or more probes. Registry construction takes probes as arguments; the registry itself performs no I/O and holds no global state at import time. To honor [ADR-0023](../../adr/0023-composition-root-ownership.md) and the incremental-probe contract in [`../rules.md`](../rules.md) ¬ß1 ("Each later wiring task appends its probe to the registry in the same PR that adds its `Container` field"), the registry **must expose a single explicit mutation primitive**:
  - `ProbeRegistry.add_probe(probe: Probe) -> None` ‚Äî appends `probe` to the registry's probe sequence in registration order. This is the **only** sanctioned mutation surface; no other mutation methods (e.g., `remove_probe`, `replace_probes`, `clear`) are added. The registry remains pure in the sense that it still performs no I/O and holds no cross-process/global state; its contents are grown exactly once, at process startup, by the wiring tasks (T-701, T-702/T-708, T-1402, T-1503) each appending exactly one probe alongside their `Container` field. Subclassing, monkeypatching, or bypassing `__slots__` (e.g., via `object.__setattr__`) to fake mutation is forbidden and is a review-BLOCKING testing anti-pattern (see [`../review.md`](../review.md)).
- An `APIRouter` builder `build_health_router(registry: ProbeRegistry, *, is_ready: Callable[[], bool]) -> APIRouter` exposing:
  - `/livez`: process check; returns `{"status":"ok"}` with **no I/O** and no probe evaluation.
  - `/healthz`: runs the injected registry's probes and returns 200 when all probes report ok, 503 otherwise.
  - `/readyz`: runs the injected registry's probes **and** consults the injected `is_ready` callable; returns 503 when either the readiness flag is false or any probe fails; 200 otherwise.
- The health module must not import from `app.core.wiring.*`, must not read `app.state`, and must not construct or reference a lifespan.

Forbidden: no direct `app.state.ready` reads, no `app.core.wiring` imports, no DB/Redis client construction here, no side effects at import time. No mutation surface on `ProbeRegistry` beyond the single `add_probe(probe)` method specified above.

Tests required

- `/livez` returns 200 with no probes registered and performs no I/O (assert via a probe registry that would raise if evaluated).
- `/healthz` returns 200 when all injected probes return ok; returns 503 when at least one probe fails.
- `/readyz` returns 503 when `is_ready()` returns `False` even if all probes pass; returns 200 when `is_ready()` returns `True` and all probes pass.
- Probe registry composition test: constructing a registry with N probes yields N evaluations in deterministic registration order.
- **`add_probe` mutation test**: starting from an empty `ProbeRegistry()`, `registry.add_probe(p1); registry.add_probe(p2)` results in `registry.probes == (p1, p2)` and `await registry.run_all()` evaluates both probes in that order. The test must exercise the real API against the real class ‚Äî no subclassing, no monkeypatching, no `object.__setattr__`.

Acceptance criteria: health module is a pure library ‚Äî endpoint shapes and probe registry unit-testable without a running FastAPI app, container, lifespan, or docker-compose. Bootstrapping the (empty) registry on the `Container` and owning the `is_ready` flag is T-504's job; each real probe is appended by its own wiring task (T-701, T-702/T-708, T-1402, T-1503); mounting the router on the app is owned by T-505; docker-compose healthcheck assertions are owned by T-507.

Commands: standard four.

Common failure modes: doing real I/O in `/livez`; reading `app.state` directly; importing `core.wiring`; declaring a lifespan hook here.

Review checklist: three endpoint shapes, probe registry pure and injectable, readiness flag injected (not read from `app.state`), no `core.wiring` import.
