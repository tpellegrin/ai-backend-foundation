## Purpose

Create the auth persistence boundary for the `users` auth fragment and `refresh_tokens` tables.

This task introduces:

- SQLAlchemy mapped classes for auth persistence.
- Immutable auth domain/read-model types required by persistence queries.
- The `0002_auth` migration.
- Integration tests proving table creation and refresh-token persistence behavior.

This task must keep SQLAlchemy ORM row types inside the persistence boundary. Query helpers must return auth domain/read-model types from `app.auth.domain`, not SQLAlchemy row objects.

The approach follows the project’s architecture rules and the Data Mapper principle: persistence code transfers data between the database representation and in-memory domain/read-model objects, keeping the persistence representation independent from the objects consumed by application logic.

## Implementation requirements

### 1. Define auth domain/read-model types

Add the minimal immutable auth domain/read-model types required by auth persistence to:

```text
app/auth/domain.py
```

These types exist so persistence can return non-ORM objects while still carrying the fields required by future auth use cases.

If existing `AuthenticatedUser` represents the post-authentication principal, do not overload it with persistence-only fields such as `password_hash`.

Add separate immutable read-model types as needed. Suggested names:

```python
@dataclass(frozen=True, slots=True)
class UserAuthRecord:
    id: UUID
    email: str
    password_hash: str
    created_at: datetime
    tenant_id: UUID | None
    disabled: bool
```

and:

```python
@dataclass(frozen=True, slots=True)
class RefreshTokenRecord:
    id: UUID
    user_id: UUID
    family_id: UUID
    hash: str
    issued_at: datetime
    expires_at: datetime
    revoked_at: datetime | None
    replaced_by: UUID | None
```

Use the project’s existing domain style. If the project already uses a naming convention different from `UserAuthRecord` or `RefreshTokenRecord`, follow that convention.

These domain/read-model types must not import SQLAlchemy.

### 2. Keep ORM rows inside persistence

Define SQLAlchemy mapped classes in:

```text
app/auth/persistence.py
```

Required mapped classes:

- `UserRow`
- `RefreshTokenRow`

These classes must inherit from the shared platform mapping foundation:

```python
from app.platform.db.base import Base
```

`app/auth/persistence.py` must not import:

```python
from app.infrastructure.db...
```

SQLAlchemy ORM row types must not cross the persistence boundary.

Public query/insert helpers should translate:

```text
UserRow -> UserAuthRecord or equivalent auth domain/read-model type
RefreshTokenRow -> RefreshTokenRecord or equivalent auth domain/read-model type
```

Do not return `UserRow` or `RefreshTokenRow` from public helper functions.

### 3. Define `UserRow`

Create a mapped `UserRow` class for the `users` table.

Required columns:

- `id`
- `email`
- `password_hash`
- `created_at`
- `tenant_id`
- `disabled`

Column requirements:

- `id` must be the primary key.
- `email` must be required.
- `password_hash` must be required.
- `created_at` must be required.
- `disabled` must be required.
- `tenant_id` may be nullable if the project’s tenancy model permits users without tenants.

Add appropriate indexes/constraints for lookup by email and tenant where required by existing task/spec conventions.

Do not bake authentication policy into persistence helpers. For example, do not silently decide that all inserted users are enabled unless the helper accepts `disabled=False` explicitly as a caller-supplied value.

### 4. Define `RefreshTokenRow`

Create a mapped `RefreshTokenRow` class for the `refresh_tokens` table.

Required columns:

- `id`
- `user_id`
- `family_id`
- `hash`
- `issued_at`
- `expires_at`
- `revoked_at`
- `replaced_by`

Column requirements:

- `id` must be the primary key.
- `user_id` must reference `users.id`.
- `family_id` must be required.
- `hash` must be required.
- `issued_at` must be required.
- `expires_at` must be required.
- `revoked_at` may be nullable.
- `replaced_by` may be nullable and should reference `refresh_tokens.id` if supported cleanly by the migration.

Add indexes required for refresh-token lookup, family-based rotation, and active-token checks.

### 5. Add family-active partial unique index

Add a family-based active-token uniqueness constraint/index.

The intended invariant is:

```text
Only one active refresh token per token family.
```

Use a PostgreSQL partial unique index equivalent to:

```sql
UNIQUE (family_id) WHERE replaced_by IS NULL
```

or, if the existing task/review convention requires the wording from earlier specs:

```text
unique on (family_id, replaced_by IS NULL)
```

Prefer the explicit PostgreSQL partial unique index form in the migration and SQLAlchemy metadata if supported by the project’s existing style.

The index should support refresh-token rotation and reuse detection without requiring ORM row leakage.

### 6. Implement persistence helper functions

Add thin persistence helper functions in `app/auth/persistence.py` as required by tests and future auth services.

These helpers should use SQLAlchemy sessions/connection objects passed by the caller. Do not create engines or sessionmakers in this module.

Helpers should:

- Insert users from explicit caller-provided values.
- Look up users by email or id as required by tests.
- Insert refresh tokens from explicit caller-provided values.
- Look up refresh tokens by hash/id/family as required by tests.
- Return auth domain/read-model types, not ORM row types.

Do not call password hashers, token generators, or auth services from persistence.

Do not decide clock values inside persistence unless the helper explicitly receives them from the caller or the task spec already defines that persistence owns those defaults.

Preferred shape for insert helpers:

```python
async def insert_user(
    session: AsyncSession,
    *,
    id: UUID,
    email: str,
    password_hash: str,
    created_at: datetime,
    tenant_id: UUID | None,
    disabled: bool,
) -> UserAuthRecord:
    ...
```

and:

```python
async def insert_refresh_token(
    session: AsyncSession,
    *,
    id: UUID,
    user_id: UUID,
    family_id: UUID,
    hash: str,
    issued_at: datetime,
    expires_at: datetime,
    revoked_at: datetime | None,
    replaced_by: UUID | None,
) -> RefreshTokenRecord:
    ...
```

Adapt names and sync/async style to the repository’s existing DB infrastructure and test style.

### 7. Add migration `0002_auth`

Create:

```text
alembic/versions/0002_auth.py
```

The migration must create:

- `users`
- `refresh_tokens`
- Required indexes
- Required foreign keys
- Family-active partial unique index

The migration must not depend on `app.auth.persistence` importing rows at migration runtime unless that is already the repository’s migration style.

Downgrade should drop created indexes/tables in a safe order. PostgreSQL drops indexes with tables, but explicit `op.drop_index(...)` calls are acceptable if they make intent clearer and match the project’s migration style.

### 8. Preserve platform DB boundary

Auth persistence may import SQLAlchemy mapping primitives only from:

```text
app.platform.db
```

For example:

```python
from app.platform.db.base import Base
```

If a mapping-time type is needed:

```python
from app.platform.db.types import ...
```

Auth persistence must not import:

```text
app.infrastructure.db
```

This task assumes T-904A has already established `app.platform.db`.

Do not recreate or modify `app.platform.db` in this task.

### 9. Test environment discipline

In `tests/integration/test_auth_persistence.py`, do not mutate environment variables with:

```python
os.environ["..."] = ...
```

Do not use module-scope environment mutation.

If environment overrides are genuinely needed, use `monkeypatch.setenv(...)` inside a function-scoped test or fixture and clear cached settings according to the existing test convention.

If tests can pass explicit mock/test settings objects directly, prefer that over environment mutation.

- Looking up the user returns the auth domain/read-model type.
- Returned value is not a SQLAlchemy row.
- `password_hash`, `email`, `tenant_id`, `disabled`, and `created_at` are preserved.

### 4. Refresh token persistence round trip

Integration test must verify:

- Inserting a refresh token succeeds.
- Looking up the token returns the auth domain/read-model type.
- Returned value is not a SQLAlchemy row.

### 5. Family-active uniqueness

Integration test must verify the family-active uniqueness rule.

At minimum:

- Two active tokens in the same family are rejected.
- A replaced token allows a new active token in the same family if that is the intended rotation model.

### 6. Boundary checks

Tests should ensure:

## Common failure modes

- Importing `Base` from `app.infrastructure.db`.
- Importing anything from `app.infrastructure.db` in `app/auth/persistence.py`.
- Returning `UserRow` or `RefreshTokenRow` from public helper functions.
- Defining domain/read-model dataclasses inside `app/auth/persistence.py`.
- Overloading `AuthenticatedUser` with `password_hash` when it represents a post-authentication principal.
- Omitting fields needed for login or refresh-token rotation from domain/read-model return types.
- Baking service policy into persistence helpers, such as silently setting `disabled=False` or `created_at=datetime.now(UTC)`.
- Creating engines or sessionmakers inside auth persistence.
- Forgetting the family-active partial unique index.
- Implementing token rotation orchestration instead of only persistence helpers.
- Mutating `os.environ` directly in task-local tests.
- Using module-scope environment mutation in integration tests.
- Implementing unrelated auth service behavior.
- Modifying `app.platform.db` in T-905 instead of relying on T-904A.

## Research rationale

The Data Mapper pattern keeps persistence representations and in-memory domain objects independent by moving data between the data store and domain representations at the persistence boundary. This supports the project rule that SQLAlchemy ORM row objects must not cross from persistence into auth service/application logic.

SQLAlchemy is commonly used as an ORM/data-mapping toolkit, and its mapped classes naturally represent database tables. In this project, those mapped classes are intentionally treated as persistence-boundary objects rather than application/domain return types.
