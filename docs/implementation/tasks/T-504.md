# Task T-504 — `app/core/container.py`, `di.py`, `lifespan.py`

> Section: **S05 — App factory + API edge**

> After you finish, self-check against [`../review.md`](../review.md) (final review checklist, architecture compliance).

Purpose: composition **library** — provide the `Container` dataclass and DI providers, and manage startup/shutdown by mutating a pre-constructed `Container` in place. The composition **site** (`create_app()`) lives in `app.main` and constructs the initial `Container` (see [ADR-0023](../../adr/0023-composition-root-ownership.md) and T-505).

Depends on: T-301, T-401–T-405.

Allowed files: `app/core/container.py`, `app/core/di.py`, `app/core/lifespan.py`, `tests/integration/test_lifespan.py`.

Forbidden: business logic; provider SDK imports outside `app/infrastructure/`.

Implementation requirements

- `Container` is an **incremental** dataclass defined in `app/core/container.py`. In T-504 it declares **only** the fields whose wiring tasks have already completed at this point: `settings: AppSettings` and `probe_registry: ProbeRegistry` (from T-405). Every later wiring task **adds** its own field to this dataclass in the same PR: T-701 appends `db_engine: AsyncEngine | None = None` and `session_factory: async_sessionmaker[AsyncSession] | None = None`; T-702/T-708 append cache/Redis fields; T-1212 appends llm+embeddings+governance fields; T-1402 appends queue; T-1503 appends vector_store. Do **not** declare placeholders for components whose wiring task has not yet run — fields land only when their adapter exists.
- All non-bootstrap fields are typed `Optional[<Protocol>]` and default to `None`; the lifespan asserts non-None after startup for the subset required by the current Phase 2 scope.
- **T-504 does not construct the `Container` or the `ProbeRegistry`.** Per [ADR-0023](../../adr/0023-composition-root-ownership.md), `create_app()` in `app.main` (T-505) builds the initial `Container` (with `settings` and an empty `ProbeRegistry`) and installs it on `app.state.container` **before** the lifespan is attached. T-504 only defines the `Container` dataclass and the lifespan/DI machinery that operates on it.
- `di.py` declares `Depends`-style providers that return entries from the `Container` stored on `app.state.container` (installed by the factory).
- `lifespan.py` is an `asynccontextmanager` that **reads the pre-constructed `Container` from `app.state.container`**, runs `await on_startup(container)` hooks, and tears down cleanly. It **must not** construct a new `Container` or `ProbeRegistry`, and **must not** reassign `app.state.container`. Fields added by later wiring tasks are mutated on the existing `Container` in place; the `ProbeRegistry` instance is never replaced. Sets `app.state.ready = True` only after all startup hooks succeed and back to `False` on shutdown.
- **Probe registry ownership.** The empty `ProbeRegistry` is constructed by `create_app()` (T-505) and lives on `container.probe_registry` for the entire process lifetime. `core.wiring` in T-504 does **not** construct a DB probe — DB infrastructure is introduced by T-701, which appends the real DB ping probe to the same registry in the PR that adds `db_engine`/`session_factory`. Redis/cache probes are appended by T-702/T-708; vector-store and queue probes are appended by T-1503 and T-1402 respectively. Each later wiring task appends its probe alongside its `Container` field, on the same registry the health router already closes over.
- **Readiness flag is owned by the lifespan here.** `app.state.ready` starts `False`; the lifespan flips it to `True` only after all startup hooks succeed and back to `False` on shutdown. The `is_ready` callable passed to T-405's `build_health_router` reads from `app.state.ready` (via a closure over the app instance); T-405 remains unaware of `app.state`.

Tests required: lifespan integration test — the `Container` is constructed by the test's `create_app()` call (not by the lifespan); `app.state.container` has the **same object identity** before startup, after startup, and after shutdown (the lifespan never replaces it); a probe appended to `container.probe_registry` **before** startup is visible on `/readyz` after startup; `app.state.ready` is `False` before startup completes and `True` after, and flips back to `False` on shutdown; `/readyz` returns 503 before startup completes and 200 after (asserted via the health router wired against the lifespan-managed flag). Do **not** assert any DB/Redis probe here — those probes are introduced by T-701/T-702/T-708.

Acceptance criteria: only `core.wiring.*` imports `app.infrastructure.*`; `Container` is the only object that knows about adapters; `lifespan.py` does not construct `Container` or `ProbeRegistry` and does not reassign `app.state.container`.

Commands: standard four.

Common failure modes: importing wiring modules at top-level of `di.py` (must be inside lifespan); constructing DB/Redis/vector/queue probes in T-504 (those belong to their own wiring tasks — T-701, T-702/T-708, T-1503, T-1402); constructing a new `Container` or `ProbeRegistry` inside `lifespan.py` (violates [ADR-0023](../../adr/0023-composition-root-ownership.md) — the factory owns construction); reassigning `app.state.container` from the lifespan.

Review checklist: Container typed, lifespan idempotent, no infra imports outside wiring.
