# Task T-904A — Establish platform SQLAlchemy mapping foundation

**Section:** S09 — Auth & Users / persistence prerequisite

Before you start, read [`../rules.md`](../rules.md) — global rules, stop signals, allowed-files discipline, command discipline, lower-complexity model instructions.

After you finish, self-check against [`../review.md`](../review.md) — final review checklist, architecture compliance.

---

## Purpose

Move the shared SQLAlchemy declarative `Base`/metadata and approved mapping-time SQLAlchemy types out of `app.infrastructure.db` and into `app.platform.db`, so feature-module persistence files can define mapped classes without importing infrastructure.

This task unblocks T-905.

This is a prerequisite architectural refactor. It does not implement auth persistence.

## Implementation requirements

### 1. Create `app.platform.db`

Create:

- `app/platform/db/__init__.py`
- `app/platform/db/base.py`

Move the shared SQLAlchemy `DeclarativeBase` and `MetaData`, including the naming convention if present, from:

```text
app/infrastructure/db/base.py
```

to:

```text
app/platform/db/base.py
```

The shared `Base` should now be imported as:

```python
from app.platform.db.base import Base
```

`app.platform.db.base` owns the shared SQLAlchemy mapping foundation. `app.infrastructure.db` must no longer own it.

### 2. Move approved mapping types

Move the `Vector` type definition from:

```text
app/infrastructure/db/types.py
```

to:

```text
app/platform/db/types.py
```

The new import path should be:

```python
from app.platform.db.types import Vector
```

Only mapping-time SQLAlchemy types belong in `app.platform.db.types`.

Runtime database helpers, engine helpers, session helpers, probes, lifecycle code, and adapter code must remain in `app.infrastructure.db`.

### 3. Update infrastructure imports

Update all valid `app.infrastructure.db` imports so infrastructure DB code imports mapping primitives from `app.platform.db`.

For example:

```python
from app.platform.db.base import Base
```

and, if needed:

```python
from app.platform.db.types import Vector
```

Delete the old infrastructure-owned files once call sites are updated:

- `app/infrastructure/db/base.py`
- `app/infrastructure/db/types.py`

Do not keep compatibility shims unless a failing test proves a short-lived shim is unavoidable. If a shim is unavoidable, stop and report before adding it.

### 4. Remove old import surface

If `app/infrastructure/db/__init__.py` currently re-exports `Base` or `Vector`, remove those re-exports.

`app.infrastructure.db` must not preserve the old mapping-foundation import path.

Repository imports should use:

```python
from app.platform.db.base import Base
```

or:

```python
from app.platform.db.types import Vector
```

for mapping primitives.

### 5. Update Alembic metadata

Update:

```text
alembic/env.py
```

so Alembic still points to the shared metadata.

For example:

```python
from app.platform.db.base import Base

target_metadata = Base.metadata
```

Adapt this to the repository’s actual Alembic structure.

The result must preserve unified metadata discovery for migrations.

### 6. Update import-linter

Update `.importlinter` only if needed to reflect the approved dependency edge:

```text
feature/module persistence files may import app.platform.db
```

Do not allow:

```text
app.<module>.persistence -> app.infrastructure.db
```

Do not add one-off import-linter exceptions.

Preserve rules preventing cross-module persistence imports.

The intended rule is:

```text
module persistence files may import app.platform.db
module persistence files must not import other modules' persistence files
module persistence files must not import app.infrastructure.db
```

### 7. Update architecture documentation

Update the following files only as needed to reflect ADR-0031 and the new module ownership:

- `docs/architecture.md`
- `docs/dependency-graph.md`
- `docs/folder-structure.md`
- `docs/phase-2-revision/02-revised-module-tree.md`
- `docs/phase-2-revision/03-revised-dependency-graph.md`

The documentation should clearly state:

- `app.platform.db` contains approved SQLAlchemy mapping primitives.
- `app.infrastructure.db` contains runtime database infrastructure.
- Feature modules may define mapped classes in local persistence files using `app.platform.db.base`.
- Feature modules must not import `app.infrastructure.db`.

If `Vector` is moved, the documentation should also clarify:

- `app.platform.db.types` contains approved SQLAlchemy mapping-time types used by module-local persistence mapping files.
- These types are not general-purpose domain/API/application dependencies.

Do not rewrite unrelated architecture sections.

### 8. Patch T-905 task spec

Update:

```text
docs/implementation/tasks/T-905.md
```

so it assumes `app.platform.db` exists.

T-905 should instruct auth persistence to import mapping primitives from:

```text
app.platform.db
```

For example:

```python
from app.platform.db.base import Base
```

T-905 must explicitly forbid auth persistence from importing:

```text
app.infrastructure.db
```

Do not implement T-905.

Do not modify `app/auth/persistence.py`.

## Acceptance criteria

- `app.platform.db.base` exists and owns the shared SQLAlchemy `Base`/metadata.
- `app.infrastructure.db` no longer owns the shared `Base`.
- `app.platform.db.types` owns `Vector` and any approved SQLAlchemy mapping-time types moved by this task.
- No imports of `app.infrastructure.db.base` remain.
- No imports of `app.infrastructure.db.types` remain.
- `app.infrastructure.db` does not preserve compatibility re-exports for the old mapping-foundation paths.
- Alembic `target_metadata` points to `Base.metadata` in the new platform location.
- No feature module imports `app.infrastructure.db`.
- T-905 is updated to point to `app.platform.db.base`.
- T-905 explicitly forbids auth persistence from importing `app.infrastructure.db`.
- Architecture docs are updated to match ADR-0031.
- Import-linter passes without one-off infrastructure exceptions.
- All standard validation commands pass.

## Common failure modes

- Adding import-linter exceptions for `app.auth.persistence -> app.infrastructure.db`.
- Adding import-linter exceptions for `app.<module>.persistence -> app.infrastructure.db`.
- Leaving `Base` owned by `app.infrastructure.db`.
- Leaving `Vector` owned by `app.infrastructure.db.types` after this task moves it.
- Preserving old forbidden import paths through compatibility re-exports.
- Moving runtime infrastructure such as engines, sessions, sessionmakers, probes, or lifecycle code into `app.platform.db`.
- Moving too many SQLAlchemy helpers into `app.platform.db`.
- Using `app.shared` as a dumping ground for database code.
- Implementing T-905 logic in this prerequisite task.
- Modifying `app/auth/persistence.py`.
- Forgetting to update `alembic/env.py`.
- Forgetting to update tests that import the old `Base` or `Vector` path.
- Forgetting to update docs or dependency graphs.
- Rewriting unrelated docs while touching architecture files.
