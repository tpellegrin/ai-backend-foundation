# Task T-906B — Fix refresh-token rotation transaction ordering

> Section: **S09 — Auth & Users / refresh-token rotation bugfix**

## Purpose

Fix the refresh-token rotation production bug exposed by T-908 integration tests.

The current refresh flow can violate either:

- the `replaced_by` foreign key, if the old token is marked as replaced before the new token exists; or
- the `ix_refresh_tokens_family_active` unique constraint, if the new active token is inserted before the old token is marked as replaced.

The fix must make refresh-token rotation atomic and compatible with the database constraints.

## Implementation requirements

- Make refresh-token rotation atomic.
- The service must not insert a new active refresh token while the old token still matches the active-token uniqueness condition.
- The service must call persistence helpers rather than manipulating ORM rows directly.
- If `replaced_by` remains a foreign key to `refresh_tokens.id`, ensure the schema and transaction ordering are compatible.
- Prefer one of these valid designs:
    - make the `replaced_by` foreign key deferrable and update old token before inserting new token within one transaction; or
    - adjust the active-token uniqueness condition so revoking the old token before inserting the new token satisfies the invariant; or
    - remove the enforced FK on `replaced_by` only if the project accepts it as a non-enforced audit/reference id.
- Preserve reuse detection: a reused/revoked/replaced token must revoke the whole family.
- Preserve hash-at-rest behavior.

## Common failure modes

- Inserting the new active token before making the old token inactive.
- Updating `replaced_by` before the replacement token exists while using an immediate FK.
- Removing the unique family-active invariant.
- Catching and ignoring `UniqueViolationError`.
- Moving SQLAlchemy logic into `service.py`.
- Fixing only the API test while leaving the DB constraint race/ordering bug.
