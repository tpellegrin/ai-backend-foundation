Purpose: bind ports to adapters per settings.

Depends on: T-704, T-702.

Allowed files: `app/core/wiring/__init__.py`, `app/core/wiring/storage.py`, `app/core/wiring/cache.py`, `app/core/container.py` (append-only), `app/core/lifespan.py` (append-only), `app/core/wiring/tests/test_wiring_storage.py`, `app/core/wiring/tests/test_wiring_cache.py`.

> T-708 is the first task that introduces the `app.core.wiring.* → app.infrastructure.*` edge. It must create `app/core/wiring/__init__.py` (empty is fine) so the package exists before any sibling wiring module (`storage.py`, `cache.py`) is imported. No `.importlinter` change is required: the existing `core-wiring-only-infra` contract already excludes `app.core.**` from its `source_modules`, and the `Layers` contract already places `app.core` above `app.infrastructure`. Do **not** weaken any import-linter contract in this task; if a future wiring task needs a new edge, that task owns the `.importlinter` update in the same diff.

Forbidden: this is the only place allowed to import `app.infrastructure.storage.*` and `app.infrastructure.redis.*`. Do not branch on `backend == "s3"` — S3 is Phase 3. Do not reorder or rewrite existing fields/hooks in `container.py` or `lifespan.py`; edits to those files are strictly append-only per rules.md §1 "Append-only edits to shared assembly files".

Implementation requirements:

- In Phase 2 only the local-FS adapter is wired (`settings.blob_storage.backend` must equal `"local"`); the wiring function returns a Protocol-typed `BlobStorage` (not the concrete class). A future Phase 3 task introduces the `"s3"` branch.
- `app/core/wiring/cache.py` builds the shared async Redis client via `app.infrastructure.redis.build_client(settings)` and constructs the `RedisCache` adapter around it. The Redis client is shared for cache, rate-limiter, and idempotency-store use.
- Append to `app/core/container.py`: a `blob_storage: BlobStorage` field and a `cache: Cache` field (Protocol-typed).
- Append to `app/core/lifespan.py`: (a) a startup hook that builds the Redis client and populates the container's `cache` (and downstream Redis-backed adapters as they land), (b) a shutdown hook that awaits `client.aclose()`, (c) a Redis readiness probe registered on the `ProbeRegistry` (per rules.md §1 incremental-probe rule). The local-FS `BlobStorage` requires no async startup/shutdown; add only its `Container` field.

Tests required: wiring picks correct adapter per backend value; container smoke test confirms the two new fields are populated with Protocol-typed instances; readiness probe for Redis is present in the registry.

Acceptance: container fields populated; Redis client lifecycle is driven exclusively by the lifespan; `lint-imports` clean.

Commands: standard four.

Common failure modes: returning the concrete adapter type as the field type; constructing a second Redis client outside the lifespan; forgetting the readiness probe.

Review checklist: typed by Protocol, no leak; container edits are strictly append-only; lifespan startup/shutdown hooks appended, not rewritten; Redis readiness probe registered.
