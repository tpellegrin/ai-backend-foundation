# Task T-906 — Auth service + policies + deps
**Section:** S09 — Auth & Users

## Purpose
Implement auth service behavior (register, login, refresh, logout) and FastAPI dependencies (`get_current_user`), orchestrating domain logic, policies, and persistence while maintaining strict boundary separation.

## Forbidden changes
- Do not modify `app/auth/persistence.py`.
- Do not modify `app/core/container.py`.
- Do not modify `app/core/wiring/*`.
- Do not modify `app/core/lifespan.py`.
- Do not add DB migrations.
- Do not use FastAPI imports in `service.py` or `policies.py`.
- Do not use SQLAlchemy imports in `service.py` or `policies.py`.
- Do not import `app.infrastructure.*` anywhere in this task.
- Do not use raw SQL or ORM rows in service.
- Do not construct `TokenSigner` or `PasswordHasher` in service/deps.
- Do not mutate `os.environ` in tests.

## Implementation requirements
- Implement `AuthService` behavior:
    - `register(email, password) -> AuthenticatedUser` (idempotent on duplicate by raising `ConflictError`).
    - `login(credentials) -> tuple[AccessToken, RefreshToken]`.
    - `refresh(refresh_token_string) -> tuple[AccessToken, RefreshToken]` — rotates and detects reuse. On detection: revoke the entire `family_id` chain using `persistence.revoke_refresh_token_family`, raise `RefreshReuseDetectedError`, and emit an audit log via `structlog`.
    - `logout(refresh_token_string) -> None`.
- Implement `policies.require_authenticated` and `require_active_user`.
- Implement `deps.get_current_user` (FastAPI dependency wrapping policies).
- Service must call persistence helpers from T-905/T-905A; it must not perform queries directly.
- Service must accept an `AsyncSession` or depend on a session-provider.
- Dependencies must retrieve already-wired `TokenSigner`, `PasswordHasher`, or `Container` fields from the approved path (e.g., `app.core.di`).
- Auth service should be instantiated via dependency injection or a request-scoped factory in `deps.py`.

## Tests required
- Service unit tests with fakes for `PasswordHasher`, `TokenSigner`, and persistence operations.
- Refresh rotation happy path (old token replaced, new one issued).
- Reuse detection (calling refresh with a replaced token) revokes the family and raises.
- Registration with existing email raises `ConflictError`.
- Login with invalid credentials raises `InvalidCredentialsError`.
- Disabled user login/refresh is rejected.
- Dependencies tests prove retrieval uses already-wired objects or fakes, not direct construction.

## Acceptance criteria
- No ORM row leakage outside persistence.
- No SQL or ORM queries in `service.py`.
- No infrastructure imports.
- No direct adapter construction in `deps.py`.
- `AuthService` correctly implements rotation and reuse detection.
- All unit tests pass.
- `make lint`, `make typecheck`, `make test` pass.

## Commands
- `make lint`
- `make typecheck`
- `make test`
- `uv run lint-imports`

## Common failure modes
- Writing SQL or ORM logic in `service.py`.
- Constructing `Argon2PasswordHasher` or `JwtSigner` directly in `deps.py`.
- Leaking `UserRow` into the service layer.
- Forgetting to revoke the entire family on refresh reuse.

## Review checklist
- Service is pure orchestration (no SQL, no SDKs).
- Refresh reuse detection revokes the full family.
- Boundaries (domain vs persistence) are respected.
