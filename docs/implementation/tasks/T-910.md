## Purpose

Implement the minimal `app.users` module and expose:

```http
GET /api/v1/users/me
```

This endpoint returns the current authenticated user’s profile.

If no profile row exists yet for the authenticated principal, the profile is created lazily on the first successful `GET /users/me` request.

This task deliberately keeps the auth persistence table and the user profile table separate:

```text
auth.users
  Owns authentication identity fields such as email, password_hash, disabled, tenant_id.

user_profiles
  Owns application user profile fields exposed by app.users.
```

`app.users` must consume the public authenticated principal returned by `get_current_user`. It must not import auth persistence, auth adapters, token signers, password hashers, or SQLAlchemy auth row types.

This task may update `.importlinter` only to authorize the narrow public API-edge dependency required for `app.users.api` or `app.users.deps` to consume public auth principal/dependency modules.

## Implementation requirements

### 1. Create the `app.users` module

Create a minimal users module under:

```text
app/users/
```

The module should contain:

```text
domain.py
persistence.py
service.py
deps.py
api.py
```

Keep responsibilities separated:

```text
app/users/domain.py
  User profile domain/read model.

app/users/persistence.py
  SQLAlchemy mapped row and persistence helpers for user_profiles.

app/users/service.py
  get_or_create_profile orchestration.

app/users/deps.py
  users-specific dependency helpers, only if needed.

app/users/api.py
  FastAPI router and request/response DTOs for /users/me.
```

### 2. Define the users domain model

Define an immutable user profile domain/read-model type in:

```text
app/users/domain.py
```

Suggested shape:

```python
@dataclass(frozen=True, slots=True)
class User:
    id: UUID
    email: str
    created_at: datetime
    updated_at: datetime
```

Use names and field types consistent with the repository’s existing style.

The `User` domain type must not import SQLAlchemy, FastAPI, auth persistence, auth adapters, or provider SDK types.

### 3. Use a separate `user_profiles` table

Create a new profile table in migration:

```text
user_profiles
```

Do not reuse or modify the existing auth `users` table in this task.

The profile table should include at least:

- `id`
- `email`
- `created_at`
- `updated_at`

The `id` value is the authenticated user id from the auth principal. It should correspond to the auth identity id, but `app.users` must not import auth persistence rows.

Add appropriate uniqueness/primary-key constraints so repeated calls for the same authenticated principal cannot create duplicate profile rows.

If a foreign key to the auth `users.id` table is added, it must be defined in the migration without introducing an application-code dependency from `app.users` to `app.auth.persistence`.

### 4. Implement users persistence

In:

```text
app/users/persistence.py
```

Define the SQLAlchemy mapped class for `user_profiles`.

The mapped class must use the shared platform mapping foundation:

```python
from app.platform.db.base import Base
```

Do not import from:

```python
app.infrastructure.db
```

Public persistence helpers must return users domain/read-model types, not SQLAlchemy row objects.

Expected helper behavior:

- Get profile by user id.
- Insert minimal profile.
- Get-or-create support primitives needed by service.

Persistence helpers should accept an existing `AsyncSession` or repository-standard session object according to the existing DB style.

Do not create engines or sessionmakers in `app.users.persistence`.

Do not commit inside low-level persistence helpers unless that is already the established project convention. Prefer `flush()` and let service/API transaction ownership follow the project’s existing pattern.

### 5. Implement `users.service.get_or_create_profile`

In:

```text
app/users/service.py
```

Implement:

```python
get_or_create_profile(user_id, *, email)
```

or the repository-style equivalent.

Behavior:

- If a profile row exists for `user_id`, return it.
- If no row exists, insert a minimal profile row using `user_id` and `email`.
- Subsequent calls return the existing row and do not duplicate it.

The service should receive the authenticated user id and email from the API/dependency layer. It must not import auth persistence, auth adapters, JWT signers, or SQLAlchemy auth row types.

### 6. Implement `GET /users/me`

In:

```text
app/users/api.py
```

Create a module router with no `/users` prefix. The `/users` prefix is applied at the mount site in `app/api/v1.py`.

Implement:

```http
GET /me
```

Effective route:

```http
GET /api/v1/users/me
```

The endpoint should:

- Retrieve the current authenticated principal using the existing auth dependency.
- Call `users.service.get_or_create_profile(user_id=current_user.<id field>, email=current_user.email)`.
- Return the user profile response DTO.

Use the existing auth dependency/current-user mechanism from the public auth surface only.

Allowed auth dependency/domain imports are limited to public API-edge/domain types such as:

```python
from app.auth.deps import get_current_user
from app.auth.domain import AuthenticatedUser
```

Do not import auth persistence, auth adapters, auth service, token internals, or SQLAlchemy auth row types.

### 7. Implement `get_authenticated_user` dependency tests

If this task introduces or uses a users-local helper such as:

```python
get_authenticated_user
```

then add unit tests for it in:

```text
app/users/tests/test_deps.py
```

The tests should verify:

- It returns the authenticated principal when present.
- It preserves the authenticated user’s id and email.
- It does not construct auth services/adapters.
- It does not decode tokens directly.
- It does not require auth persistence imports.

If no users-local `get_authenticated_user` helper is introduced, test the equivalent users API dependency behavior in `app/users/tests/test_api.py`.

### 8. Lazy profile creation path

Profile creation in Phase 2 uses the lazy-read path:

```python
users.service.get_or_create_profile(user_id=current_user.<id field>, email=current_user.email)
```

This is invoked from `users.api` on the first successful `GET /users/me` for the authenticated principal.

If no `user_profiles` row exists, the service inserts a minimal profile record inside the same request transaction and returns it.

Subsequent calls read the existing row and must not duplicate it.

This preserves the dependency-graph constraint:

```text
auth -> users remains forbidden
```

without introducing cross-module events or a user-profile writer port.

## Tests required

Add tests covering:

### API behavior

- `GET /api/v1/users/me` returns `401` when unauthenticated.
- `GET /api/v1/users/me` returns `200` when authenticated.
- Response body contains the current user’s profile.
- Response includes the current authenticated user’s email.
- Problem Details shape is preserved for unauthenticated access if the existing auth/API error handling uses Problem Details.

### Lazy creation

- First `GET /users/me` for a freshly registered/authenticated principal creates a `user_profiles` row lazily.
- Second `GET /users/me` for the same principal returns the same profile row.
- Second call does not create a duplicate profile.

### Service behavior

- `get_or_create_profile(user_id, email=...)` returns existing profile when present.
- `get_or_create_profile(user_id, email=...)` creates a minimal profile when absent.
- The service does not import auth persistence or auth adapters.

### Dependency behavior

- Unit tests cover `get_authenticated_user` or the equivalent users-auth dependency path.
- The dependency path preserves both authenticated user id and email.
- The dependency path does not decode JWTs directly.

### Persistence behavior

- Persistence helpers return `app.users.domain.User`, not SQLAlchemy row objects.
- `user_profiles` uniqueness/primary-key constraints prevent duplicate rows for the same authenticated user id.

### Migration behavior

- `0003_users.py` creates `user_profiles`.
- Migration imports pass Ruff sorting rules.
- Migration can be applied in integration tests using the repository’s existing migration/test pattern.

### Import-boundary behavior

- `uv run lint-imports` passes.
- The import-linter change allows only the public users API/dependency edge to auth dependency/domain modules.
- `app.users.domain`, `app.users.service`, and `app.users.persistence` remain independent from `app.auth`.
- No `app.users.*` module imports auth persistence, auth adapters, auth service, or token internals.
