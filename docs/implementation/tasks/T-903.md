## Purpose

Implement the `PasswordHasher` port with Argon2id and introduce the missing Argon2 configuration required to make password hashing tunable by environment.

This task is the first consumer of password-hashing settings, so it owns adding the Argon2 settings block to `AppSettings` and updating the `PasswordHasher` `Protocol` with `needs_rehash`.

This task also owns updating existing auth-port tests and fake password hashers that directly exercise the `PasswordHasher` contract.

Argon2id is the intended algorithm because current password-storage guidance recommends Argon2id for password hashing, and the `argon2-cffi` high-level `PasswordHasher` uses Argon2id by default.

## Implementation requirements

### 1. Update the `PasswordHasher` Protocol

Update `PasswordHasher` in `app/auth/ports.py` to expose:

```python
def needs_rehash(self, password_hash: str) -> bool: ...
```

The port should support:

```python
def hash(self, password: str) -> str: ...
def verify(self, password: str, password_hash: str) -> bool: ...
def needs_rehash(self, password_hash: str) -> bool: ...
```

Prefer normalizing the project-level port signature to:

```python
def verify(self, password: str, password_hash: str) -> bool: ...
```

This order is the public application contract. The adapter is responsible for adapting that order to the provider API.

If existing tests or fakes use the previous argument order, update only the direct auth-port test/fake implementations allowed by this task.

Do not update unrelated auth behavior or unrelated tests.

Preserve the boolean-returning `verify(...) -> bool` contract unless the existing project code already defines a different contract.

The hasher should answer whether a plaintext password matches a stored hash. Authentication/application-flow code owns translating a failed verification into `InvalidCredentialsError`.

Therefore, if the port contract remains `verify(...) -> bool`, the Argon2 adapter must return `False` on mismatch rather than raising `InvalidCredentialsError`.

`argon2-cffi.PasswordHasher.verify(...)` returns `True` on success and raises provider-specific exceptions such as `VerifyMismatchError`, `VerificationError`, or `InvalidHashError` on failure. The adapter must translate those provider behaviors into the public port contract.

### 2. Update existing auth-port tests and fakes

Update `app/auth/tests/test_auth_ports.py` only as needed to keep the direct `PasswordHasher` port tests aligned with the final port contract.

If the task normalizes the signature to:

```python
def verify(self, password: str, password_hash: str) -> bool: ...
```

then any `FakePasswordHasher` or direct test helper in `app/auth/tests/test_auth_ports.py` must be updated to use the same method signature.

This permission is limited to the `PasswordHasher` contract change introduced by T-903.

Do not use this task to change unrelated auth ports, unrelated auth fake behavior, or unrelated auth tests.

### 3. Add Argon2 settings

Add an `Argon2Settings` block to `app/core/config/settings.py` and wire it into `AppSettings`.

Include:

- `time_cost`
- `memory_cost`
- `parallelism`
- `hash_len`
- `salt_len`

Use positive integer validation for all five settings.

Use Argon2id.

Use `argon2.PasswordHasher` and pass the configured values into it. The high-level API exposes `time_cost`, `memory_cost`, `parallelism`, `hash_len`, `salt_len`, and `type`.

Recommended production-oriented defaults:

```python
time_cost = 2
memory_cost = 19456
parallelism = 1
hash_len = 32
salt_len = 16
```

Rationale:

- OWASP lists Argon2id `m=19456`, `t=2`, `p=1` as one of its recommended configurations and summarizes this as a minimum Argon2id configuration of 19 MiB memory, 2 iterations, and 1 degree of parallelism.
- `argon2-cffi` documents that `memory_cost` is measured in KiB and that `salt_len` of 16 bytes is sufficient for all applications.
- RFC 9106 defines Argon2 password-hashing inputs including parallelism, memory size in KiB, number of passes, salt length, and tag length; it recommends 16-byte salts for password hashing.

Do not treat these values as untouchable constants. They are defaults that must remain configurable by environment.

### 4. Preserve Pydantic settings compatibility

Add `Argon2Settings` using the existing project settings pattern.

The Argon2 settings block must support the projectŌĆÖs existing nested environment-variable convention, including double-underscore overrides such as:

```env
ARGON2__MEMORY_COST=19456
```

Prefer a nested settings submodel under the root `AppSettings` if that matches the existing Pydantic v2 settings style.

A typical acceptable shape is:

```python
    memory_cost: int = Field(default=19456, gt=0)

```

and:

```python

```

on the root `AppSettings`, adjusted to match the projectŌĆÖs actual settings conventions.

Do not introduce a separate nested `BaseSettings` if it breaks nested environment-variable parsing or test initialization.

Do not make required Argon2 settings impossible to instantiate in tests without environment variables unless that is already the established project settings pattern.

### 7. Implement `Argon2PasswordHasher`

Implement `Argon2PasswordHasher` in `app/auth/adapters/argon2_hasher.py`.

It must structurally satisfy `PasswordHasher`.

Required behavior:

- Use `argon2.PasswordHasher`.
- Use Argon2id explicitly.
- Use configured settings for Argon2 parameters.
- Implement `hash(password: str) -> str`.
- Implement `verify(password: str, password_hash: str) -> bool`.
- Implement `needs_rehash(password_hash: str) -> bool`.
- Return `True` for valid password/hash pairs.
- Return `False` for password mismatch.
- Return `False` or otherwise safely handle malformed hashes according to the established auth-port contract.
- Do not leak provider-specific Argon2 exceptions through the public port.

Pay attention to argument order: the project port should expose `verify(password, password_hash)`, while `argon2-cffi.PasswordHasher.verify(...)` expects the encoded hash first and the plaintext password second.

### 8. Implement rehash detection

Implement `needs_rehash(password_hash: str) -> bool` using `argon2.PasswordHasher.check_needs_rehash(...)`.

`argon2-cffi` documents `check_needs_rehash(...)` as the way to determine whether a stored hash was created using the current hasher parameters and recommends checking after successful authentication so outdated hashes can be upgraded at the next opportunity.

The adapter method name should remain `needs_rehash(...)` to keep the project port independent of the provider API name.

### 9. Configuration boundary

Prefer passing a small Argon2 config object or explicit primitive values into `Argon2PasswordHasher`.

Good shape:

```python
Argon2PasswordHasher(
    time_cost=settings.argon2.time_cost,
    memory_cost=settings.argon2.memory_cost,
    parallelism=settings.argon2.parallelism,
    hash_len=settings.argon2.hash_len,
    salt_len=settings.argon2.salt_len,
)
```

Or:

```python
Argon2PasswordHasher(settings.argon2)
```

Only use this second form if `settings.argon2` is a small dedicated config object.

Avoid:

```python
Argon2PasswordHasher(settings: AppSettings)
```

unless the task is explicitly updated to allow the adapter to depend on the global application settings type.

## External rationale

- OWASP recommends Argon2id for password storage and lists `m=19456`, `t=2`, `p=1` as a recommended Argon2id configuration.
- `argon2-cffi.PasswordHasher` uses Argon2id by default, supports configurable `time_cost`, `memory_cost`, `parallelism`, `hash_len`, and `salt_len`, and exposes `check_needs_rehash(...)`.
- `argon2-cffi` recommends measuring parameters on the deployment environment because appropriate hashing cost depends on available memory, CPU, and user-experience budget.
- RFC 9106 defines Argon2 inputs including memory size, pass count, parallelism, salt length, and tag length, and recommends 16-byte salts for password hashing.
