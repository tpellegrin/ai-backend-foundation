# Task T-505 — `app/main/app_factory.py` `create_app()`

> Section: **S05 — App factory + API edge**
>
> Before you start, read [`../rules.md`](../rules.md) (global rules, stop signals, allowed-files discipline, command discipline, lower-complexity model instructions).
> After you finish, self-check against [`../review.md`](../review.md) (final review checklist, architecture compliance).

---


Purpose: composition **site** — produce a fully configured FastAPI app for serving and for tests, and construct the initial `Container` (see [ADR-0023](../../adr/0023-composition-root-ownership.md)).

Depends on: T-501..T-504.

Allowed files: `app/main/app_factory.py`, `app/main/tests/test_app_factory.py`. (The `app/main/__init__.py` ASGI binding `app = create_app()` is owned by T-506.)

> **No `__init__.py` in test directories.** Per [`../rules.md`](../rules.md) §4, test directories under `app/<module>/tests/` and `tests/` must not carry an `__init__.py` marker; doing so packages them and collides with pytest's rootdir-based import (empirically breaks collection of the top-level `tests/` package). `app/main/tests/` follows the same rule as every other module's test directory. If a prior revision of this task listed `app/main/tests/__init__.py` in `Allowed files`, that entry is retracted.

Rationale for the location: `create_app()` must import from both `app.api` (routers, exception handlers) and `app.core` (lifespan, Container, DI). Only `app.main` sits at a layer that is allowed to import both under the Layers contract in `importlinter.toml`. Placing the factory in `app.core` would require an `app.core → app.api` edge and is rejected — the `.importlinter` Layers contract is **not** to be weakened.

Forbidden: adding `ignore_imports` entries to `.importlinter`; constructing a second `Container` or `ProbeRegistry` anywhere in the process; reassigning `app.state.container` from anywhere other than `create_app()`.

Implementation requirements

- `create_app(settings: AppSettings | None = None) -> FastAPI`:
  - resolves `settings` (falling back to `get_settings()` from T-301) and configures logging (via T-401),
  - **constructs the initial `Container`** (from `app.core.container`) with `settings` and an **empty** `ProbeRegistry` (from T-405 — an empty registry whose contents are later grown by wiring tasks via `ProbeRegistry.add_probe(probe)`); this is the sole construction site for both objects for the process lifetime,
  - creates the `FastAPI` instance and **assigns `app.state.container = container` before attaching the lifespan**, so the health router and any downstream code that reads `app.state.container` see the same object the lifespan will later mutate in place,
  - installs middleware in this exact order (outermost first): `CorrelationMiddleware` (T-402) → `AccessLogMiddleware` (T-404) → `SecurityHeadersMiddleware` (T-502; construct with `settings.api.security_headers_enabled`) → `starlette.middleware.cors.CORSMiddleware` configured **from the `Api` settings group** (T-301: `cors_allowed_origins`, `cors_allow_credentials`, `cors_allowed_methods`, `cors_allowed_headers`),
  - mounts the `/api/v1` router from T-503 (`build_v1_router()`) and the health router from T-405 (`build_health_router(registry=container.probe_registry, is_ready=lambda: app.state.ready)`); the health router therefore closes over the **same `ProbeRegistry` instance** that later wiring tasks (T-701, T-702/T-708, T-1402, T-1503) will mutate via `probe_registry.add_probe(probe)`,
  - registers exception handlers via `register_exception_handlers` (T-501),
  - sets the lifespan from `app.core.lifespan` (T-504), which **reads and mutates** `app.state.container` in place and owns `app.state.ready`; the lifespan must not construct a new `Container` or `ProbeRegistry` and must not reassign `app.state.container`.
- CORS is registered here — not in T-502. When `cors_allowed_origins` is empty, `CORSMiddleware` is still installed but denies every cross-origin request (deny-by-default).
- The ASGI binding `app = create_app()` in `app/main/__init__.py` is the ASGI entrypoint (`app.main:app`) and is added by T-506, not T-505.

Tests required

- App boots in-process; `/api/v1` and health routes are registered.
- **Container ownership**: `create_app()` returns an app whose `app.state.container` is populated **before** the lifespan runs; the object's identity is unchanged after startup and after shutdown (the lifespan never replaces it).
- **Shared `ProbeRegistry`**: after `create_app()` returns, calling `app.state.container.probe_registry.add_probe(probe)` is observed by `/readyz` once the lifespan has run — i.e., the health router closes over the same registry instance later wiring tasks mutate. The test **must** exercise the real `ProbeRegistry.add_probe` API introduced by T-405; monkeypatching `ProbeRegistry`, subclassing it, or bypassing `__slots__` via `object.__setattr__` is a testing anti-pattern and constitutes a review-BLOCKING defect (see `../review.md`).
- **Middleware registered once**: no middleware class appears more than once in `app.user_middleware`.
- **CORS deny-by-default**: with `cors_allowed_origins=()`, a cross-origin preflight from any origin does not receive `Access-Control-Allow-Origin`.
- **CORS whitelist**: with `cors_allowed_origins=("https://ok.example",)`, a preflight from `https://ok.example` succeeds and any other origin is denied.
- **Security headers**: every response carries all five headers listed in T-502.
- **Readiness semantics**: `/readyz` returns 503 before lifespan startup completes and 200 after; `/livez` returns 200 in both states.

Acceptance criteria: `app.main:app = create_app()`; middleware, CORS, and both routers are registered exactly once from this task; `create_app()` is the sole construction site for `Container` and `ProbeRegistry`; `.importlinter` is unchanged (no `ignore_imports` entries added).

Commands: standard four.

Common failure modes: building two `FastAPI` instances; middleware order wrong (correlation must wrap access log); constructing a second `Container` or `ProbeRegistry` in the lifespan (violates [ADR-0023](../../adr/0023-composition-root-ownership.md), leaves the health router closed over an orphaned registry); placing the factory in `app.core` (would require an `app.core → app.api` edge, forbidden by the Layers contract); adding `ignore_imports` to `.importlinter` instead of fixing the layer edge.

Review checklist: middleware order, lifespan attached, no duplicate handlers.
